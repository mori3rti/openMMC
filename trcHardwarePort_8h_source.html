00069 /******************************************************************************
00070  * Supported ports
00071  *
00072  * PORT_HWIndependent
00073  * A hardware independent fallback option for event timestamping. Provides low
00074  * resolution timestamps based on the OS tick.
00075  * This may be used on the Win32 port, but may also be used on embedded hardware
00076  * platforms. All time durations will be truncated to the OS tick frequency,
00077  * typically 1 KHz. This means that a task or ISR that executes in less than
00078  * 1 ms get an execution time of zero.
00079  *
00080  * PORT_APPLICATION_DEFINED
00081  * Allows for defining the port macros in other source code files.
00082  *
00083  * PORT_Win32
00084  * "Accurate" timestamping based on the Windows performance counter for Win32
00085  * builds. Note that this gives the host machine time, not the kernel time.
00086  *
00087  * Hardware specific ports

00091  * external developers. They should work, otherwise let us know by emailing
00092  * support@percepio.com. Some work on any OS platform, while other are specific
00093  * to a certain operating system.

00119 /*******************************************************************************
00120  * IRQ_PRIORITY_ORDER
00121  *
00122  * Macro which should be defined as an integer of 0 or 1.
00123  *
00124  * This should be 0 if lower IRQ priority values implies higher priority
00125  * levels, such as on ARM Cortex M. If the opposite scheme is used, i.e.,
00126  * if higher IRQ priority values means higher priority, this should be 1.
00127  *
00128  * This setting is not critical. It is used only to sort and colorize the
00129  * interrupts in priority order, in case you record interrupts using
00130  * the vTraceStoreISRBegin and vTraceStoreISREnd routines.
00131  *
00132  ******************************************************************************
00133  *
00134  * HWTC Macros
00135  *
00136  * These four HWTC macros provides a hardware isolation layer representing a
00137  * generic hardware timer/counter used for driving the operating system tick,
00138  * such as the SysTick feature of ARM Cortex M3/M4, or the PIT of the Atmel
00139  * AT91SAM7X.
00140  *
00141  * HWTC_COUNT: The current value of the counter. This is expected to be reset
00142  * a each tick interrupt. Thus, when the tick handler starts, the counter has
00143  * already wrapped.
00144  *
00145  * HWTC_COUNT_DIRECTION: Should be one of:
00146  * - DIRECTION_INCREMENTING - for hardware timer/counters of incrementing type
00147  *  such as the PIT on Atmel AT91SAM7X.
00148  *  When the counter value reach HWTC_PERIOD, it is reset to zero and the
00149  *  interrupt is signaled.
00150  * - DIRECTION_DECREMENTING - for hardware timer/counters of decrementing type
00151  *  such as the SysTick on ARM Cortex M3/M4 chips.
00152  *  When the counter value reach 0, it is reset to HWTC_PERIOD and the
00153  *  interrupt is signaled.
00154  *
00155  * HWTC_PERIOD: The number of increments or decrements of HWTC_COUNT between
00156  * two OS tick interrupts. This should preferably be mapped to the reload
00157  * register of the hardware timer, to make it more portable between chips in the
00158  * same family. The macro should in most cases be (reload register + 1).
00159  * For FreeRTOS, this can in most cases be defined as
00160  * #define HWTC_PERIOD (configCPU_CLOCK_HZ / configTICK_RATE_HZ)
00161  *
00162  * HWTC_DIVISOR: If the timer frequency is very high, like on the Cortex M chips
00163  * (where the SysTick runs at the core clock frequency), the "differential
00164  * timestamping" used in the recorder will more frequently insert extra XTS
00165  * events to store the timestamps, which increases the event buffer usage.
00166  * In such cases, to reduce the number of XTS events and thereby get longer
00167  * traces, you use HWTC_DIVISOR to scale down the timestamps and frequency.
00168  * Assuming a OS tick rate of 1 KHz, it is suggested to keep the effective timer
00169  * frequency below 65 MHz to avoid an excessive amount of XTS events. Thus, a
00170  * Cortex M chip running at 72 MHZ should use a HWTC_DIVISOR of 2, while a
00171  * faster chip require a higher HWTC_DIVISOR value.
00172  *
00173  * The HWTC macros and vTracePortGetTimeStamp is the main porting issue
00174  * or the trace recorder library. Typically you should not need to change
00175  * the code of vTracePortGetTimeStamp if using the HWTC macros.
00176  *
00177  ******************************************************************************/
