00314 /*******************************************************************************
00315  * xTraceOpenLabel
00316  *
00317  * Creates user event labels for user event channels or for individual events.
00318  * User events can be used to log application events and data for display in
00319  * the visualization tool. A user event is identified by a label, i.e., a string,
00320  * which is stored in the recorder's symbol table.
00321  * When logging a user event, a numeric handle (reference) to this string is
00322  * used to identify the event. This is obtained by calling
00323  *
00324  *   xTraceOpenLabel()
00325  *
00326  * whihc adds the string to the symbol table (if not already present)
00327  * and returns the corresponding handle.
00328  *
00329  * This can be used in two ways:
00330  *
00331  * 1. The handle is looked up every time, when storing the user event.
00332  *
00333  * Example:
00334  *   vTraceUserEvent(xTraceOpenLabel("MyUserEvent"));
00335  *
00336  * 2. The label is registered just once, with the handle stored in an
00337  * application variable - much like using a file handle.
00338  *
00339  * Example:
00340  *   myEventHandle = xTraceOpenLabel("MyUserEvent");
00341  *   ...
00342  *   vTraceUserEvent(myEventHandle);
00343  *
00344  * The second option is faster since no lookup is required on each event, and
00345  * therefore recommended for user events that are frequently
00346  * executed and/or located in time-critical code. The lookup operation is
00347  * however fairly fast due to the design of the symbol table.
00348  ******************************************************************************/
00349 traceLabel xTraceOpenLabel(const char* label);

00361  /******************************************************************************
00362  * vTracePrintF
00363  *
00364  * Advanced user events (Professional Edition only)
00365  *
00366  * Generates User Event with formatted text and data, similar to a "printf".
00367  * It is very fast compared to a normal "printf" since this function only
00368  * stores the arguments. The actual formatting is done
00369  * on the host PC when the trace is displayed in the viewer tool.
00370  *
00371  * User Event labels are created using xTraceOpenLabel.
00372  * Example:
00373  *
00374  *   traceLabel adc_uechannel = xTraceOpenLabel("ADC User Events");
00375  *   ...
00376  *   vTracePrint(adc_uechannel,
00377  *               "ADC channel %d: %lf volts",
00378  *               ch, (double)adc_reading/(double)scale);
00379  *
00380  * This can be combined into one line, if desired, but this is slower:
00381  *
00382  *   vTracePrint(xTraceOpenLabel("ADC User Events"),
00383  *               "ADC channel %d: %lf volts",
00384  *               ch, (double)adc_reading/(double)scale);
00385  *
00386  * Calling xTraceOpenLabel multiple times will not create duplicate entries, but
00387  * it is of course faster to just do it once, and then keep the handle for later
00388  * use. If you don´t have any data arguments, only a text label/string, it is
00389  * better to use vTraceUserEvent - it is faster.
00390  *
00391  * Format specifiers supported:
00392  * %d - 32 bit signed integer
00393  * %u - 32 bit unsigned integer
00394  * %f - 32 bit float
00395  * %s - string (is copied to the recorder symbol table)
00396  * %hd - 16 bit signed integer
00397  * %hu - 16 bit unsigned integer
00398  * %bd - 8 bit signed integer
00399  * %bu - 8 bit unsigned integer
00400  * %lf - double-precision float (Note! See below...)
00401  *
00402  * Up to 15 data arguments are allowed, with a total size of maximum 32 byte.
00403  * In case this is exceeded, the user event is changed into an error message.
00404  *
00405  * The data is stored in trace buffer, and is packed to allow storing multiple
00406  * smaller data entries in the same 4-byte record, e.g., four 8-bit values.
00407  * A string requires two bytes, as the symbol table is limited to 64K. Storing
00408  * a double (%lf) uses two records, so this is quite costly. Use float (%f)
00409  * unless the higher precision is really necessary.
00410  *
00411  * Note that the double-precision float (%lf) assumes a 64 bit double
00412  * representation. This does not seem to be the case on e.g. PIC24 and PIC32.
00413  * Before using a %lf argument on a 16-bit MCU, please verify that
00414  * "sizeof(double)" actually gives 8 as expected. If not, use %f instead.
00415  ******************************************************************************/
00416 void vTracePrintF(traceLabel eventLabel, const char* formatStr, ...);
